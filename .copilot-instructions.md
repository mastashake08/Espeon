# Espeon - Heltec LoRa 32 V3 Multi-Partition Project

## Hardware Platform
- **Board**: Heltec WiFi LoRa 32 V3
- **MCU**: ESP32-S3FN8 (8MB Flash)
- **Key Features**: 
  - Built-in LoRa radio (SX1262)
  - 0.96" OLED display (128x64)
  - WiFi & Bluetooth
  - PRG button (GPIO 0)
  - Built-in battery management

## Project Architecture

### Multi-Partition System
This project uses ESP32 OTA partition scheme with multiple application partitions:
- **Factory partition**: Contains the menu application (bootloader default)
- **Additional partitions**: Host separate applications that can be loaded dynamically
- Each partition contains a standalone binary application

### Factory Menu Application
The factory application serves as a partition manager and menu system:
- Displays available applications on the OLED screen
- Handles user navigation using the PRG button
- Implements partition switching via ESP32 OTA API
- Always accessible as the fallback application

### PRG Button Navigation
- **Single click**: Navigate through menu items/partitions
- **Double click**: Load and boot the selected partition's binary
- **Long press** (optional): Return to factory menu from any app

## Development Guidelines

### Code Structure
- Keep each application self-contained within its partition
- Factory app should maintain a partition table/registry
- Use ESP32's `esp_ota_*` functions for partition management
- Implement button debouncing for reliable input detection

### Display Management
- Initialize the OLED display in each application
- Factory menu should show: partition name, description, and navigation hints
- Consider battery status display
- Use clear visual indicators for selected items

### Button Handling
```cpp
// PRG button is typically GPIO 0
#define PRG_BUTTON 0
// Implement single-click, double-click, and long-press detection
// Use interrupts or polling with proper debouncing
```

### Partition Management
- Use `partition.csv` to define custom partition table
- Each app partition should have descriptive labels
- Factory app needs access to partition metadata
- Implement safe partition switching with validation

### Dependencies
- Heltec ESP32 Dev-Boards library (for display and board support)
- ESP32 Arduino framework
- Consider OneButton library for advanced button handling

## Best Practices
1. Always validate partition data before switching
2. Implement watchdog timers to prevent boot loops
3. Store partition metadata in NVS (Non-Volatile Storage)
4. Test button handling thoroughly (debouncing is critical)
5. Provide visual feedback for all user actions
6. Include error handling for failed partition loads
7. Consider adding a serial console menu for debugging

## Common Tasks

### Creating a New Application Partition
1. Define partition in `partition.csv`
2. Create separate PlatformIO environment
3. Build binary for specific partition
4. Update factory menu to include new app

### OTA Partition Switching
```cpp
#include "esp_ota_ops.h"
#include "esp_partition.h"

// Find and boot target partition
const esp_partition_t* partition = esp_partition_find_first(
    ESP_PARTITION_TYPE_APP, 
    ESP_PARTITION_SUBTYPE_ANY, 
    "app_name"
);
esp_ota_set_boot_partition(partition);
esp_restart();
```

### Button Event Detection
- Implement state machine for click detection
- Track press duration for single/double/long press
- Use millis() for timing, not delay()
- Add visual feedback (LED/display) on button press

## Naming Conventions
- Partition labels: lowercase with underscores (e.g., `factory`, `app_lora`, `app_wifi`)
- Application descriptive names in menu (e.g., "LoRa Messenger", "WiFi Scanner")
- Source files: organized by application in separate directories
